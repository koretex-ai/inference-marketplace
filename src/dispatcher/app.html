How it works

Koretex Points

Points reward the providers who make the network great: nodes that stay online, run quality hardware, and serve the best, genuine models — designed so being a good provider is the only way to climb. Your standing also influences how much real traffic routes to your nodes.

What earns points

🟢 Uptime & reliability

We continuously health-check every node with real workloads. Consistent availability and completed work counts — not just being switched on.

⚡ Hardware quality

We measure what your machine delivers — throughput under load — not a spec sheet. Faster hardware that serves bigger models well earns more, with tapering returns.

🧠 Model authenticity

Serving in-demand models earns more, and we verify a node really runs the model it claims. Swapping in a cheaper model is detected and penalised.

How it fits together

points += uptime × ( hardware + model value ) × trust − penalties
  • Uptime sets your earning rate — points accumulate the whole time you're up & serving. The more uptime, the faster they add up; an outage just pauses earning — it never takes back points you've already banked.
  • Hardware + model value add up — a strong machine, plus serving authentic models people want, earns more per unit of uptime.
  • Trust ramps with time — new nodes start limited and grow into full standing, so throwaway nodes are pointless.
  • Penalties subtract — a fake model or repeated failed checks cost you; repeat offenders are removed.

How each part is calculated

Beyond the high-level formula, here's exactly what drives each term — and the live constants we use today.

Uptime — your earning rate

Every minute your node is up & serving (verified by unannounced health checks ~1/min) banks an uptime credit. Points accrue against those credits, so more uptime means you earn faster — and going offline only pauses earning, it never deducts points you've already banked.

Hardware — measured throughput

From the tokens/sec your node actually generates under load (not your spec sheet), curved as √(tok/s) so returns taper — going 20→80 tok/s is about 2× the hardware term, not 4×. Bigger, faster Macs win, just sub-linearly. A faster engine (e.g. MLX) measures higher here.

Model value — demand × model size × diversity × authenticity

√(tokens you served × model weight), so it scales with real demand and with how heavy the model is — a token on a big model is worth more than one on a small model (heavy models are scarcer and cost far more memory). The weight runs ~0.5× for a 3B up to ~4.4× for a 235B; see each model's ×pts on the Models tab. +10% per extra in-demand model you serve. If a health check catches you serving a different model than you claim, this collapses to 5%.

Trust — the ramp (10% → 100%)

New nodes start at 10% weight and ramp to full over ~14 days of clean history. This makes mass-producing throwaway nodes pointless — standing is earned over time.

Per-operator diminishing returns

Across one account's nodes, the best counts full, the 2nd ~85%, the 3rd ~72%… so running more good hardware still helps, just sub-linearly. One large farm can't crowd everyone out.

Penalties

Serving a fake model or repeatedly failing checks subtracts points directly; persistent offenders are removed from the network and hidden from the board.

How we measure it

  • Live health checks. Roughly every minute the network sends your node a real inference request — identical to a customer's — measuring reachability, throughput, and whether it's truly running the claimed model.
  • Verifying the model. Checks use a fixed prompt with deterministic settings; nodes running the same model produce the same answer. Disagree, and you're flagged.
  • Credited to you. Every request is routed to one node and tracked by a unique id it returns — so attribution is exact, never guessed.

This system will evolve 🔧

Points are provisional and will change as we tune each signal against real data. Every contribution is recorded in a permanent history, so when the methodology improves we can re-evaluate fairly. The principles are stable — reward uptime, hardware, and genuine models; make gaming pointless — but the details will keep getting sharper.

Earn from your computer

Run a node

Rent your idle GPU/Mac time to run AI inference — hardware-agnostic: Apple Silicon Macs, NVIDIA GPUs (Linux or Windows via WSL2), even CPU. You earn Koretex Points for the work your node does, with withdrawal coming in the future. No hardware is favored — demand flows to whoever is fastest/cheapest/best for each request.

How it works

  1. Connect your account — it's your node's identity and where your points accrue.
  2. Copy the one-line command below and run it in your terminal (see your platform's steps).
  3. Your machine auto-starts, serves requests, and earns. Track it on your dashboard.

Works on: Apple Silicon Mac (16GB+), Linux/Windows + NVIDIA GPU, or CPU-only (slow, low demand). Your account's secret never leaves your device — you only sign a message to prove ownership.

It installs everything (engine, model, auto-start via launchd — survives reboot) and links your wallet. If it stops asking for Node.js, install it once from nodejs.org (macOS LTS .pkg) and re-run. Needs macOS 13+, 16GB+.

🪟 Windows + NVIDIA — via WSL2 (Ubuntu)

You'll run the node inside WSL2, a real Ubuntu Linux that uses your NVIDIA GPU. Do everything in the Ubuntu window — not PowerShell (PowerShell gives 'DISPATCHER=…' is not recognized). First make sure your NVIDIA driver is up to date (from nvidia.com / GeForce Experience) — that's what exposes the GPU to WSL.

  1. Open an admin terminal: right-click the Start button → Terminal (Admin).
  2. Install Ubuntu: wsl --install, then reboot. After reboot an Ubuntu window opens and asks you to create a username + password (used for sudo). If none opens, launch Ubuntu from the Start menu.
  3. Turn on systemd (keeps the node running) — in the Ubuntu window:
    printf '[boot]\nsystemd=true\n' | sudo tee /etc/wsl.conf
    Then in PowerShell run wsl --shutdown, and reopen Ubuntu from the Start menu.
  4. Check the GPU is visible from Ubuntu: nvidia-smi should list your RTX card.
  5. Install Node 20 + zstd:
    sudo apt-get update && sudo apt-get install -y zstd curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash - && sudo apt-get install -y nodejs
  6. Paste your install command (from step 1 above) into the Ubuntu window. It sets the node up as a system service that survives logout. (To survive a full Windows reboot too, see "Keep your node running" below.)

🐧 Linux + NVIDIA (or CPU)

  1. Make sure your NVIDIA driver is installed (nvidia-smi works).
  2. Install Node 20 + zstd:
    sudo apt-get update && sudo apt-get install -y zstd curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash - && sudo apt-get install -y nodejs
    (RHEL/Fedora: sudo dnf install -y zstd nodejs · Arch: sudo pacman -S zstd nodejs — Node must be 20+.)
  3. Paste the install command from step 1. It installs as a systemd system service (survives logout/reboot) — nothing else to do.

Control your node

Manage your node with the koretex command. If it says command not found, open a new terminal, or run export PATH="$HOME/.local/bin:$PATH" (older installs: $HOME/.koretex/bin) — see troubleshooting.

koretex statusShow your wallet link, whether you're serving, and which models are loaded.
koretex modelsInteractive model picker — shows every model that fits your hardware with its ×pts multiplier and $/1M, best-earner first, and pulls the ones you choose.
koretex models lsList what you're serving, what else your machine can run, and what needs more accelerator memory — each with its ×pts and $/1M.
koretex models add <tag>Pull and start serving one or more models, e.g. koretex models add qwen3:14b gpt-oss:20b. Live as soon as the download finishes. (Pulls from the public Ollama registry — see troubleshooting for custom models.)
koretex models rm <tag>Stop serving and delete one or more models to free up disk.
koretex stop / koretex startPause / resume serving (banked points stay — downtime never deducts them).
koretex pairRe-link this machine to a wallet. Run again only to move the node to a different wallet.

More models served = more demand you can earn from, plus a diversity bonus. Pick the highest ×pts models your hardware can comfortably run.

Keep your node running

By default the node runs under your login session, so it stops when you log out or close the terminal. To keep it serving 24/7:

macOSAlready persistent — the launchd service auto-starts at login and respawns on crash. Nothing to do.
Linux / WSLInstalls as a systemd system service (runs under PID 1 as your user) — already persistent across logout, no linger needed. Control: sudo systemctl status|start|stop koretex-node-agent. (Older --user installs instead need sudo loginctl enable-linger $USER.)
Windows / WSL — always-onWSL also shuts the Linux VM down when idle or on reboot. To survive that, create a Windows Task Scheduler task that runs wsl -d Ubuntu -u root true (or wsl ~) At log on / At startup — that boots the distro (and, with linger enabled, your node) automatically.

Bring it back manually anytime: koretex start (or systemctl --user start koretex-ollama koretex-node-agent).

Troubleshooting

koretex: command not foundThe koretex wrapper isn't on your current shell's PATH yet. Open a new terminal, or run export PATH="$HOME/.local/bin:$PATH" now (older installs used $HOME/.koretex/bin — use that path if the file is there: ls ~/.koretex/bin/koretex). New terminals pick it up automatically.
'DISPATCHER=…' is not recognized (Windows)You're in PowerShell. The installer is a Linux/bash script — run it inside WSL2 Ubuntu instead (see Windows setup above).
Node stops when I close the terminalLinger isn't enabled. Run sudo loginctl enable-linger $USER. On Windows/WSL also set up the Task Scheduler entry above so the WSL VM itself stays/comes back up.
Failed to connect to bus (systemctl --user, WSL)Your per-user systemd manager isn't up. First confirm systemd is running: ps -p 1 -o comm= should print systemd — if not, enable it (sudo sh -c 'printf "[boot]\nsystemd=true\n" > /etc/wsl.conf', then wsl --shutdown in PowerShell and reopen). Then start the user manager: sudo loginctl enable-linger $USER, sudo systemctl start user@$(id -u).service, and export XDG_RUNTIME_DIR=/run/user/$(id -u). Verify with ls /run/user/$(id -u)/bus; now systemctl --user … works.
systemctl: not found / -ash prompt (WSL)You're in the wrong distro (e.g. docker-desktop). Enter Ubuntu: wsl -d Ubuntu (and wsl --set-default Ubuntu to make it the default).
Stuck at 2/5 Node.js…You need Node 20+. curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash - && sudo apt-get install -y nodejs, then re-run the install command.
This engine needs 'zstd' at 3/5sudo apt-get install -y zstd (or dnf/pacman), then re-run. Re-runs are safe.
It's slow / not using my GPUCheck the engine is on the GPU, not CPU: while serving, nvidia-smi should list an ollama process. If a model is too big for your VRAM it falls back to CPU — pick a smaller model with koretex models.
"I already have the model in my Ollama"The node serves through its own managed engine (port 11435), separate from your personal Ollama (11434) and from a Windows Ollama. Add models with koretex models add <tag> (registry models only). A custom model isn't in the registry — copy its files into the managed engine's store, or recreate it there.
Install fails downloading /agent.js (404)Transient during a dispatcher redeploy. Wait a couple of minutes and re-run the install command.
Check what's happeningkoretex status · logs: journalctl --user -u koretex-node-agent -f (Linux/WSL) or /tmp/koretex-agent.log & /tmp/koretex-ollama.log.
